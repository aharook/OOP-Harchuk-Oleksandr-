# Code Smells та рефакторинг: практичний аналіз

### 1. Знайдені запахи коду та рефакторинг

### Порушення єдиної відповідальності (God Method)
    Опис: Метод ProcessOrder робив абсолютно все: валідував суму, міняв статус, виводив текст та імітував email.

    Техніка: Виділення класів (Extract Class).

    До: Уся логіка містилася в єдиному класі OrderProcessor.

    Після: Створено вузькоспеціалізовані класи: OrderValidator для перевірок, EmailService для листів, OrderRepository для збереження та OrderService для їх координації.

### Жорстка зв'язність (Tight Coupling)

    Опис: Обробник замовлень напряму залежав від виводу в консоль (std::cout). Додати запис у БД чи справжній email без зміни ядра було б неможливо.

    Техніка: Впровадження залежностей (Dependency Injection).

    До: Усі дії були жорстко зашиті в код обробника.

    Після: Виділено абстрактні інтерфейси (IOrderValidator, IEmailService), які передаються в OrderService через конструктор.

### Магічні числа (Magic Numbers / Inconsistent Logic)

    Опис: У коді була логічна суперечність: перевірка суми вимагала > 0, але текст помилки казав "більше ніж 1".

    Техніка: Інкапсуляція бізнес-правил.

    До: Умови та повідомлення містили хардкод різних чисел.

    Після: Перевірку суми надійно сховано в методі IsValid() класу SimpleOrderValidator.

### 2. Чому рефакторинг без тестів є ризикованим

Ризик непомітних поломок
    Змінюючи код наосліп, легко зламати існуючі правила. Наприклад, виправляючи плутанину з числами > 0 та > 1, розробник міг випадково змінити саму логіку. Без тестів програма збереться без помилок, але почне блокувати валідні замовлення клієнтів.

Перевага тестів
    Модульні тести миттєво підсвічують такі проблеми. Якби існував тест для замовлення на суму рівно 1.0, він би одразу провалився під час рефакторингу, сигналізуючи про зміну поведінки системи ще до того, як код потрапить у використання.
